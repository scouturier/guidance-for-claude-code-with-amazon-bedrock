# LLM Gateway Integration

This guide explains how to integrate this guidance with an existing LLM gateway (such as LiteLLM) so that Claude Code traffic flows through your organization's centralized proxy instead of connecting to Amazon Bedrock directly.

## When to Use This

This integration path is intended for organizations that **already operate an LLM gateway** and want to route Claude Code through it rather than deploying the full monitoring stack included in this guidance.

If you do not already have an LLM gateway, the native monitoring stack (OpenTelemetry + CloudWatch) deployed by this guidance already provides:

- Per-user token usage and cost attribution
- Usage quotas with SNS alerts
- Historical analytics via Athena
- CloudTrail audit trails with user attribution

There is no functional gap that requires a gateway. Use this guide only if your organization has a pre-existing gateway investment to leverage.

## What Changes (and What Doesn't)

Routing through an LLM gateway only affects **how Claude Code reaches Bedrock**. Everything else remains unchanged:

| Concern | Impact |
|---|---|
| OIDC authentication | Unchanged — users still authenticate via your IdP |
| IAM temporary credentials | Unchanged — per-user session-tagged credentials are still issued |
| CloudTrail attribution | Unchanged — provided the gateway forwards user credentials to Bedrock (see [Credential Handling](#credential-handling)) |
| OTel monitoring | Unchanged — metrics flow directly from Claude Code to your collector, independent of the Bedrock routing path |
| Package distribution | Unchanged — `ccwb package` still builds and distributes installers |

## Architecture

```
Developer Machine
  │
  ├── OIDC auth (IdP) ──► IAM temp credentials (per-user, session-tagged)
  │
  └── Claude Code
        │  ANTHROPIC_BEDROCK_BASE_URL
        ▼
    LLM Gateway  ◄── your organization's existing gateway
        │
        ▼
    Amazon Bedrock
```

## Gateway Solutions

Any gateway that exposes a Bedrock-compatible or Anthropic Messages-compatible endpoint can be used. For a self-hosted option, [LiteLLM](https://github.com/BerriAI/litellm) is the most commonly deployed open-source gateway and is referenced in the [Claude Code upstream docs](https://code.claude.com/docs/en/llm-gateway).

For commercial and managed options, browse available solutions on [AWS Marketplace](https://aws.amazon.com/marketplace/search/?searchTerms=llm+gateway).

## Configuration

### Step 1: Deploy This Guidance Normally

Follow the [Quick Start](../../QUICK_START.md) through `ccwb deploy`. The authentication infrastructure (IAM OIDC Provider or Cognito Identity Pool, IAM roles) must be deployed regardless of whether a gateway is used.

Monitoring infrastructure (`ccwb deploy monitoring`) is optional — if your gateway already provides the observability you need, you can skip it.

### Step 2: Configure the Gateway Endpoint

The gateway must expose a Bedrock-compatible endpoint. Set the following environment variables in `~/.claude/settings.json` (or in the package settings generated by `ccwb package`):

```json
{
  "env": {
    "CLAUDE_CODE_USE_BEDROCK": "1",
    "AWS_PROFILE": "ClaudeCode",
    "ANTHROPIC_BEDROCK_BASE_URL": "https://your-gateway-host/bedrock-proxy",
    "CLAUDE_CODE_SKIP_BEDROCK_AUTH": "1"
  }
}
```

`CLAUDE_CODE_SKIP_BEDROCK_AUTH=1` tells Claude Code not to sign Bedrock requests itself — the gateway is responsible for forwarding them to Bedrock with appropriate credentials.

> **Note:** If your gateway exposes an Anthropic Messages API format (e.g., `/v1/messages`) rather than a Bedrock format, use `ANTHROPIC_BASE_URL` instead of `ANTHROPIC_BEDROCK_BASE_URL`, and omit `CLAUDE_CODE_USE_BEDROCK`.

#### LiteLLM example

**`litellm_config.yaml`** — configure LiteLLM to route to Bedrock and preserve the headers Claude Code depends on:

```yaml
model_list:
  - model_name: claude-sonnet
    litellm_params:
      model: bedrock/us.anthropic.claude-sonnet-4-6-20251101-v1:0
      aws_region_name: us-east-1

general_settings:
  master_key: your-litellm-master-key   # clients send this as Authorization: Bearer
  forward_client_headers_to_llm_provider: true  # preserves anthropic-beta, anthropic-version
```

LiteLLM uses the IAM role attached to the host (EC2, ECS, Lambda) or the standard AWS credential chain to authenticate to Bedrock. No static credentials are needed if the host has an appropriate IAM role.

**`~/.claude/settings.json`** — point Claude Code at LiteLLM using the Anthropic Messages format (recommended, simpler):

```json
{
  "env": {
    "ANTHROPIC_BASE_URL": "https://your-litellm-host:4000",
    "ANTHROPIC_AUTH_TOKEN": "your-litellm-master-key"
  }
}
```

Or keep the Bedrock format if your gateway exposes a Bedrock-compatible endpoint:

```json
{
  "env": {
    "CLAUDE_CODE_USE_BEDROCK": "1",
    "AWS_PROFILE": "ClaudeCode",
    "ANTHROPIC_BEDROCK_BASE_URL": "https://your-litellm-host:4000/bedrock",
    "CLAUDE_CODE_SKIP_BEDROCK_AUTH": "1",
    "ANTHROPIC_AUTH_TOKEN": "your-litellm-master-key"
  }
}
```

### Step 3: Configure Gateway Authentication

Claude Code sends an `Authorization` header with each request. Two options are supported:

**Static token:**

```bash
export ANTHROPIC_AUTH_TOKEN=your-gateway-token
```

Or in `settings.json`:

```json
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "your-gateway-token"
  }
}
```

**Dynamic token via helper script** (recommended for short-lived tokens or vault-backed credentials):

```json
{
  "apiKeyHelper": "/path/to/your/token-helper-script"
}
```

The helper script must print a valid token to stdout. Set `CLAUDE_CODE_API_KEY_HELPER_TTL_MS` to control how often it is called (e.g., `3600000` for hourly refresh).

## Credential Handling

How the gateway authenticates to Bedrock determines whether per-user CloudTrail attribution is preserved.

**Credential pass-through (recommended):** The gateway accepts the user's AWS SigV4-signed request and forwards it to Bedrock using the user's credentials. CloudTrail records the user's IAM identity and session tags on every Bedrock API call — attribution is fully preserved.

**Gateway-owned credentials:** The gateway uses its own IAM role to call Bedrock. CloudTrail records the gateway's IAM identity, not the individual user's. Per-user attribution in CloudTrail is lost, though OTel monitoring (if enabled) continues to provide user-level metrics at the application layer.

Choose the approach that meets your organization's audit requirements.

## Gateway Requirements

Your gateway must faithfully forward the following, or Claude Code features will degrade:

| Header / Field | Required by |
|---|---|
| `anthropic-beta` header | Beta features (extended thinking, etc.) |
| `anthropic-version` header | API version negotiation |
| `anthropic_beta` body field | Bedrock-format requests |
| `X-Claude-Code-Session-Id` | Session-level aggregation in gateway logs |

If your gateway sits in front of a Bedrock-format endpoint and strips or rewrites these, set `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1` to prevent feature-detection mismatches.

## Troubleshooting

**Claude Code fails to authenticate to the gateway**

Verify `ANTHROPIC_AUTH_TOKEN` is set or `apiKeyHelper` returns a non-empty token. Check that the gateway accepts the `Authorization: Bearer <token>` header format.

**Requests reach the gateway but fail at Bedrock**

Confirm the gateway is forwarding the `anthropic-version` and `anthropic-beta` headers. Confirm the IAM role used by the gateway has `bedrock:InvokeModel` and `bedrock:InvokeModelWithResponseStream` permissions in the target regions.

**Beta features (extended thinking, etc.) stop working**

Your gateway may be stripping the `anthropic-beta` header. Either configure the gateway to preserve it, or set `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1` to disable features that depend on it.
